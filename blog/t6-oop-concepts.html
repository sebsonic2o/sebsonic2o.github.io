A Story of Composition
Or thinking about Object-Oriented "versus" Functional Programming...

I have been exposed to Object-Oriented Programming for some time now. So I was very interested to look into Functional Programming as well. At first, I was surprised by the amount of debate over those two paradigms. It feels like OOP wizards want to defend their turf at all cost against FP wizards (and vice versa). With a fresh outlook, a lot of it seems to be fueled by ignorance or close-mindedness.

Both programming concepts leverage data and operations on data. The difference boils down to how we choose to compose the code and what discipline we want to impose.

In OOP, the code is organized with objects that can hold data (state), and a contract is formed with functions that can be used with the object (behavior). Such a contract might look a little bit like this:

  • I am a kind of rectangle
  • Width and height differentiate me from other rectangles
  • You can change me to a square


      class Rectangle
        def initialize(width, height)
          @width = width
          @height = height
        end

        def to_square
          if @width > @height
            @height = @width
          else
            @width = @height
          end
        end
      end

In FP, the code is organized with functions, and a contract is signed on data in and out of the function. Such a contract might look a little bit like this:

  • I am a function to square
  • You can give me a structure with width and height as input
  • I will not change the input
  • I will output a new structure with same width and height
  • Given the same input, I will always return the same output


      Rectangle = Struct.new(:width, :height)

      def to_square(rectangle)
        if rectangle.width > rectangle.height
          Rectangle.new(rectangle.width, rectangle.width)
        else
          Rectangle.new(rectangle.height, rectangle.height)
        end
      end

To get to an end result with FP, you will then chain functions without changing the state of variables. This means that there is no concurrent update problems in your program since nothing is getting updated.

So are FP and OOP mutually exclusive? Not really. FP principles are great for reducing data mutation concerns, whereas OOP principles are excellent for defining architectural boundaries. For instance, I am glad that I can use FP features such as recursions, lambdas, or procs in my OO programs written in Ruby.

Fortunately, many others in the community do share the same view. We see functional and object-oriented programming as valid paradigms that enhance software development when used wisely. It is like salt and pepper. You may need a little bit of both for your meal to taste good!
